PVC Distributor Admin — End-to-End Flow

The whole journey: invite a user → stock in → sell → deliver → invoice → get paid → report. Plus the live cloud setup.

🔄 1. Master business flow (everything, end to end)

Follow the arrows top-to-bottom. Green = money/stock in, red = problem paths, amber = automatic background jobs.

flowchart TD
    %% ===== ONBOARDING =====
    subgraph ONB[👋 Onboarding]
      O1[Admin invites user] --> O2[User sets password via email link] --> O3[Login → Dashboard]
    end

    %% ===== MASTER DATA =====
    subgraph MAS[🧱 Set up master data]
      M1[Categories → Products → SKUs/Dimensions]
      M2[Suppliers]
      M3["Customers (+ 📱 phone)"]
    end
    O3 --> M1 & M2 & M3

    %% ===== PROCURE TO PAY =====
    subgraph P2P[🏭 Buy stock — Procure-to-Pay]
      PA[Restock Cart] --> PB[Purchase Order] --> PC{Approve} --> PD[Goods Receipt]
      PD --> PE[Supplier Invoice] --> PF[💸 Supplier Payment]
    end
    M2 --> PA
    PD -->|stock IN ➕| STK[(📦 Stock Ledger)]

    %% ===== ORDER TO CASH =====
    subgraph O2C[🛒 Sell — Order-to-Cash]
      SA[Quotation optional] --> SB[Sales Order]
      SB --> SC{Credit limit & stock OK?}
      SC -->|no| SX[⛔ Blocked / on-hold / backorder]
      SC -->|yes| SD[Confirm order]
      SD --> SE[🚚 Delivery — dispatch]
      SE --> SF[🧾 Invoice — issue]
      SF --> SG{Payment}
      SG -->|full| SH[✅ Paid]
      SG -->|partial| SI[🟡 Partially paid]
      SG -->|not by due date| SJ[⏰ Overdue]
      SI --> SG
    end
    M1 --> SB
    M3 --> SA
    SE -->|stock OUT ➖| STK
    SG --> LED[(📒 Customer Ledger)]
    SE -->|goods returned| RET[↩️ Return → Credit Note]
    SJ --> REM["📱 WhatsApp reminder
(single or Remind-All-Overdue)"]

    %% ===== REPORTING =====
    SH --> RPT[📊 Reports & Dashboard]
    LED --> RPT
    STK --> RPT

    %% ===== AUTOMATIC ALERTS =====
    STK -.daily 8 AM.-> AL[⏰ Alerts cron]
    SJ -.daily 8 AM.-> AL
    AL --> NTF[🔔 Notifications to Admin]
stock / money IN blocked / overdue / return automatic (cron) core document step
Everything ties to a customer (orders, invoices, ledger, reminders) and to a SKU (stock in/out). Recording a payment is what flips an invoice to Paid / Partially Paid and posts to the ledger.

🛒 2. "How an order becomes cash" (the critical path)

sequenceDiagram
    participant C as Customer
    participant S as Sales staff
    participant W as Warehouse
    participant A as Accounts
    S->>S: New Order (pick customer, add products)
    S->>S: System checks credit limit + on-hold
    S->>W: Order confirmed
    W->>W: Delivery dispatch (stock deducted)
    W->>A: Goods delivered
    A->>C: Invoice issued (PDF)
    C->>A: Pays (full / partial / cheque)
    A->>A: Record Payment → status updates, ledger credited
    A-->>C: If overdue → WhatsApp reminder

📦 3. Stock movement (single source of truth)

Current quantity of any SKU = the sum of all its ledger movements.

flowchart LR
    GR[Goods Receipt] -->|+ qty| L[(Stock Ledger)]
    OB[Opening Balance] -->|+ qty| L
    ADJ[Adjustment +/-] --> L
    DMG[Damage] -->|- qty| L
    DSP[Delivery dispatch] -->|- qty| L
    RET[Return restock] -->|+ qty| L
    L --> NOW[Current qty per SKU]
    NOW --> LOW{Below threshold?}
    LOW -->|yes| AL[🔔 Low-stock alert]

☁️ 4. Live deployment (how it runs in the cloud)

UI on Vercel, API + database on Render. A request and its login token flow like this:

flowchart LR
    U[🌐 Browser] -->|https| V["Vercel
pvc-admin.vercel.app
(Next.js UI)"]
    V -->|"/api/v1 calls
Bearer token + cookie"| R["Render
pvc-backend...onrender.com
(Express API)"]
    R --> DB[("Render PostgreSQL
pvc_admin")]
    R -->|SMTP| MX[✉ Email — Mailtrap/Gmail]
    V -.->|wa.me link| WA[📱 WhatsApp]
    R -.->|daily 8 AM| CRON[⏰ Alerts]
Because UI and API are on different domains, the login refresh cookie is SameSite=None; Secure so sessions persist. CORS on the API is locked to the Vercel URL.

🚀 5. Update → redeploy flow

flowchart TD
    DEV[You make a change] --> Q{What changed?}
    Q -->|Backend code| GP[git push to GitHub]
    Q -->|DB schema| MIG[Add migration file → git push]
    Q -->|Frontend code| FE{Frontend deploy}

    GP --> RND[Render auto-builds & redeploys ✅]
    MIG --> RND
    RND --> RUN[On boot: npm run migrate
applies new migrations]

    FE -->|Option A| CLI["Re-run: npx vercel deploy --prod"]
    FE -->|Option B| CONN[Connect Vercel ↔ GitHub once
→ then auto-deploys on push]
    CLI --> LIVEF[New UI live]
    CONN --> LIVEF
Backend & DB = automatic on git push Frontend = CLI command, or connect Git for auto
Env-var changes: Render redeploys automatically; on Vercel you must redeploy after changing NEXT_PUBLIC_API_URL (it's baked in at build time). Never run npm run seed on the live DB — it wipes & recreates users/roles.

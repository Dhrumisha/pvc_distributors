PVC Distributor Admin — System Guide

A complete, visual walkthrough of every module and how work flows through the system — from inviting a user to getting paid.

🧭 1. What this system is

A distributor management system for a PVC / building-materials business. It runs the full cycle: buy stock from suppliers, hold inventory, sell to customers, deliver, invoice, collect payment, and report on it all — with role-based access so each staff member sees only what they should.

🛡️ 4. Roles & access (RBAC)

Access is controlled per module × action. There are 10 modules and 6 actions each (view, create, edit, delete, approve, export) = 60 permissions. Roles bundle permissions; users get roles.

flowchart LR
    USER[👤 User] -->|has| ROLE[🎖️ Role]
    ROLE -->|granted| PERM["🔑 Permission
(module.action)"]
    PERM --> GATE{Allowed?}
    GATE -->|yes| OK[✅ See page / do action]
    GATE -->|no| NO[🚫 Hidden / blocked]
    ADMIN[👑 Admin role] -.bypasses all checks.-> OK
RoleTypical access
AdminEverything (bypasses all checks). Manages users, roles, settings.
Sales StaffCustomers, quotations, sales orders; view inventory/invoices/payments.
Warehouse StaffInventory, stock, deliveries; view sales orders/suppliers.
AccountantInvoices, payments, reports; view customers/suppliers.
ViewerRead-only across all modules.

Manage these at Admin → Roles (create roles, toggle each permission) and Admin → Users (assign roles).

👋 5. New user onboarding & login

The system is invite-only — nobody can self-sign-up. An Admin invites a person; they set their own password from an emailed link.

    S->>S: Activate account (status ACTIVE)
    N->>S: /auth/login (email + password)
    S-->>N: Token + redirect to 🏠 Dashboard

Forgot / reset password

flowchart LR
    F[Login → "Forgot password?"] --> G[Enter email]
    G --> H[✉ Reset link emailed]
    H --> I[/auth/reset-password/]
    I --> J[Set new password] --> K[✅ Login]
Where: Invite at Admin → Users → Invite User. If the email didn't arrive, use the Resend Invite action on that user. Emails require SMTP to be configured (see §13).

🔐 6. Authentication & token refresh

Login returns a short-lived access token (15 min) used on every API call, plus a long-lived refresh token stored in a secure cookie (7 days). When the access token expires, the app silently refreshes it — so you stay logged in without re-typing your password.

sequenceDiagram
    participant B as Browser (Admin UI)
    participant API as Backend
    B->>API: POST /auth/login (email, password)
    API-->>B: access token (15m) + refresh cookie (7d)
    B->>API: any request + Bearer access token
    Note over B,API: ...15 minutes pass, token expires...
    B->>API: request → 401 Unauthorized
    B->>API: POST /auth/refresh (refresh cookie)
    API-->>B: fresh access token
    B->>API: retry original request ✅

🧱 8. Products & stock setup (master data)

Before selling, set up what you stock. A Product (e.g. "ACP Sheet") has one or more SKUs / dimensions (e.g. 4mm × 5×20). Stock is tracked per SKU through the Stock Ledger.

flowchart LR
    CAT[Categories] --> PROD[Products]
    PROD --> DIM["SKUs / Dimensions
(size, selling price)"]
    DIM --> STK[(Stock Ledger)]
    GR[Goods Receipt] -->|stock IN ➕| STK
    DSP[Delivery dispatch] -->|stock OUT ➖| STK
    ADJ[Adjustment / Damage] --> STK
    STK --> LOW{Below threshold?}
    LOW -->|yes| ALERT[🔔 Low-stock alert]

Related screens: Inventory → Categories / Products / Stock Ledger / Batch & Lots / Restock Cart. Every stock change (receipt, dispatch, adjustment, damage) writes a line to the Stock Ledger, so current quantity is always the sum of its movements.

🏭 9. Procure-to-Pay (buying stock)

How stock comes in and how you pay suppliers.

flowchart LR
    RC[📥 Restock Cart
auto-suggest low items] --> PO[Purchase Order]
    PO --> AP{Approve}
    AP -->|approved| GR[Goods Receipt
record what arrived]
    GR -->|stock IN ➕| STK[(Stock Ledger)]
    GR --> SI[Supplier Invoice
bill from supplier]
    SI --> SP[💸 Supplier Payment]
    SP --> DONE[✅ Supplier paid]
  1. Restock Cart — auto-suggests items below threshold; convert the cart into a Purchase Order.
  2. Purchase Order (Suppliers → Purchase Orders) — what you're ordering from which supplier; gets approved.
  3. Goods Receipt — record what physically arrived → adds stock (and logs damage if any).
  4. Supplier Invoice — the bill the supplier sends you.
  5. Supplier Payment (Payments → Supplier Payments) — money you pay out.

▶️ 11. How to get (create) an order — step by step

1
Go to Sales → Sales Orders → "New Order".
2
Pick the customer. Their phone shows instantly (or warns if missing). If the customer is On Hold or would exceed their credit limit, the order is blocked.
3
Choose Delivery Type (our vehicle / customer vehicle) and an optional required date.
4
Add products: select a Product → its SKU/dimension (the unit price auto-fills from the SKU), set quantity, GST %, discount %. Add more lines as needed.
5
Watch the live totals (subtotal, GST, grand total) update.
6
Click Create Order → it opens the new order's detail page (status draft).
7
From the order: Confirm it, then proceed to Delivery → Invoice → Payment.
An order is always tied to one customer, so it automatically reuses that customer's phone for any later WhatsApp reminders — no need to re-enter a number.

💳 12. Payments & receivables (who owes you)

The hub is Billing → Invoices — it shows due date, total, paid, balance and status, with an Overdue Only filter.

flowchart LR
    INV[🧾 Issued invoice] --> P{Record Payment}
    P -->|full balance| PAID[Status: Paid ✅]
    P -->|part| PART[Status: Partially Paid]
    P -->|nothing by due date| OD[Status: Overdue ⏰]
    PART --> P
    OD --> P
    P --> LED[(Customer Ledger
credit entry)]
    OD --> REM[📱 Remind / Remind All Overdue]

Recording a payment (this is how you update status)

  1. Open Billing → Invoices (or an invoice's detail page).
  2. Click the ₹ Record Payment action on any unpaid invoice.
  3. Enter amount (defaults to full balance), mode (cash/UPI/bank/cheque/card), date, reference.
  4. Full balance → Paid; less → Partially Paid. It also posts a credit to the customer's ledger automatically.

Also under Payments: Customer Payments (all receipts), Supplier Payments (what you pay out), Aging Report (0–30/31–60/61–90/90+ buckets), Cheques Due (upcoming cheque dates).

Configuring SMTP

Set these in pvc-backend/.env then restart the backend:

  • Gmail (real delivery): SMTP_HOST=smtp.gmail.com, SMTP_PORT=587, SMTP_USER=you@gmail.com, SMTP_PASS=<16-char App Password>. Needs 2-Step Verification + an App Password (not your normal password).
  • Mailtrap (testing only): catches emails in a sandbox inbox; does not deliver to real inboxes.
On startup the backend logs ✓ SMTP ready when it connected, or a warning if not. If email isn't configured, invites/resets won't send.

📱 14. WhatsApp payment reminders

Free, click-to-send reminders for pending payments. It opens WhatsApp with the customer's number and a ready-made message — you tap Send.

flowchart LR
    A[Customer has 📱 phone saved] --> B[Customer has an UNPAID invoice]
    B --> C{Where you click}
    C -->|per invoice| D[Green 💬 on Invoices row
or invoice detail "Remind"]
    C -->|in bulk| E["Remind All Overdue"
guided queue]
    D --> F[WhatsApp opens with
pre-filled message]
    E --> F
    F --> G[You tap Send ✅]
Prerequisites: the green WhatsApp button only appears on unpaid invoices (issued / partially paid / overdue), and only works if the customer has a phone number saved. A brand-new customer with no invoice has nothing to remind yet.

To verify: set a customer's phone to a number you can check → open an unpaid invoice for them → click 💬 → confirm the message arrives. (Use a recipient different from the WhatsApp account you're sending from.)

⏰ 15. Automatic alerts (scheduled job)

Once a day (default 8 AM) a background job runs three checks and creates in-app notifications for Admins:

flowchart LR
    CRON[⏰ Daily 8 AM] --> L[Low-stock check]
    CRON --> O[Overdue invoices → mark Overdue]
    CRON --> C[Cheques due soon]
    L --> NTF[🔔 Notifications]
    O --> NTF
    C --> NTF

Configured via ALERT_CRON_SCHEDULE. The job is wrapped so a failure can never crash the API.

🔔 17. Notifications & audit logs

Notifications

Bell icon in the top bar + the Notifications page. Shows low-stock, overdue, and cheque alerts. Mark one or all as read.

Audit Logs Admin

Admin → Audit Logs records who did what (create/edit/delete), on which module, when, and from which IP. Filter by action/module.

📖 20. Glossary

TermMeaning
SKU / DimensionA specific variant of a product (size/spec) that stock is counted against.
Stock LedgerThe list of every stock movement; current quantity = sum of movements.
Goods ReceiptRecording stock that physically arrived from a supplier (adds stock).
Sales OrderA confirmed customer order, before delivery/invoicing.
InvoiceThe bill sent to a customer; tracks total, paid, balance, status.
Credit NoteA credit issued to a customer (e.g. for a return).
LedgerA running account of what a customer was billed vs what they paid.
AgingHow long invoices have been unpaid, bucketed (0–30/31–60/61–90/90+ days).
RBACRole-Based Access Control — permissions grouped into roles given to users.
On HoldA customer flagged so no new orders can be created for them.
